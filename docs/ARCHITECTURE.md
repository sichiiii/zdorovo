# Архитектура

Zdorovo состоит из трёх небольших процессов. Они общаются через D-Bus и файлы в
пользовательском каталоге данных.

## Основное приложение

`healthbreak.py` содержит планировщик, SQLite-хранилище и GTK-интерфейс. Один
экземпляр приложения владеет именем `io.github.jabka.Zdorovo` в session bus.
Закрытие окна скрывает интерфейс, но не останавливает процесс.

Планировщик получает снимок активности, обновляет экранное время и накопленные
таймеры. Запись в SQLite идёт только из главного потока. Сбор активности AT-SPI
вынесен в отдельный поток, чтобы обход accessibility-дерева не блокировал UI.

## Расширение GNOME

`extension/extension.js` раз в секунду записывает идентификатор активного
приложения, время бездействия, полноэкранный статус и наличие screen sharing в
`~/.local/share/zdorovo/activity.json`.

Расширение не читает заголовки окон и не показывает собственный интерфейс.
Если heartbeat устарел, приложение автоматически переходит на AT-SPI и
`org.gnome.Mutter.IdleMonitor`.

## Индикатор

`indicator.py` использует Ayatana AppIndicator и отправляет действия основному
приложению через `org.gtk.Actions`. Индикатор не открывает второй экземпляр
приложения.

## Хранилище

SQLite работает в WAL-режиме. В базе находятся интервалы активности, события
напоминаний, дневные отметки самочувствия, привычки, открытые достижения и
локальный центр уведомлений. Прогресс достижений вычисляется из событий, а
момент открытия сохраняется отдельно, поэтому эмблема не теряется при изменении
текущей статистики. Настройки хранятся в JSON и записываются атомарной заменой
файла.

## Жизненный цикл

Пользовательские unit-файлы systemd запускают основное приложение и индикатор
вместе с графической сессией. `launch.sh` сначала поднимает сервисы, затем
активирует уже существующий GTK-процесс через D-Bus.
